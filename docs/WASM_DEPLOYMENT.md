## Building the WASM Application

Before deploying, the Rust frontend must be compiled to WebAssembly. There are two main approaches.

### Prerequisites

```bash
# Install the WASM target
rustup target add wasm32-unknown-unknown
```

### Option A: Trunk (Recommended)

[Trunk](https://trunkrs.dev/) is the recommended build tool for eframe/egui WASM applications. It handles asset bundling, wasm-bindgen glue generation, and dev server in one tool.

**Install:**

```bash
cargo install trunk
```

**Develop locally:**

```bash
# Serves at http://localhost:8080 with hot-reload
trunk serve frontend/index.html
```

**Build for production:**

```bash
# Outputs to frontend/dist/
trunk build frontend/index.html --release
```

The `frontend/dist/` directory will contain:
- `index.html` — entry page (auto-patched to load WASM)
- `ensemble-frontend-<hash>_bg.wasm` — the compiled WebAssembly binary
- `ensemble-frontend-<hash>.js` — JavaScript glue code generated by wasm-bindgen
- Any other static assets referenced in `index.html`

These files are everything needed for deployment — just serve them as static files.

### Option B: wasm-pack + Manual Integration

[wasm-pack](https://rustwasm.github.io/wasm-pack/) gives more control and is useful if you want to integrate the WASM module into a custom JavaScript/TypeScript project.

**Install:**

```bash
cargo install wasm-pack
```

**Build:**

```bash
# Build targeting the web (generates ES module output)
wasm-pack build frontend --target web --out-dir pkg
```

This produces a `frontend/pkg/` directory containing:
- `ensemble_frontend_bg.wasm` — compiled WebAssembly
- `ensemble_frontend.js` — ES module JavaScript bindings
- `ensemble_frontend.d.ts` — TypeScript type definitions
- `package.json` — npm package metadata

**Manual HTML integration:**

You then need to create your own HTML page that loads the module:

```html
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="utf-8" />
    <title>Ensemble Emulator</title>
    <style>
        html, body { margin: 0; padding: 0; width: 100%; height: 100%; overflow: hidden; }
    </style>
</head>
<body>
    <script type="module">
        import init from './pkg/ensemble_frontend.js';
        init();
    </script>
</body>
</html>
```

> **Note:** The eframe `WebRunner` in the app creates its own `<canvas>` element and appends it to `<body>`, so no canvas element is needed in the HTML.

## Deployment Options

All options below serve the build output (`dist/` or `pkg/` + HTML) as static files. No server-side code is needed.

### 1. GitHub Pages

**Best for:** Free hosting directly from the repository. Zero-cost, tight integration with CI.

**Setup with GitHub Actions:**

Create `.github/workflows/deploy-wasm.yml`:

```yaml
name: Deploy WASM to GitHub Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Rust
        uses: dtolnay/rust-toolchain@nightly
        with:
          targets: wasm32-unknown-unknown

      - name: Install Trunk
        run: cargo install trunk

      - name: Build WASM
        run: trunk build frontend/index.html --release --public-url .

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: frontend/dist

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

Then enable GitHub Pages in **Settings → Pages → Source → GitHub Actions**.

The app will be available at `https://<username>.github.io/<repo-name>/`.

> **Tip:** Use `--public-url .` with Trunk so assets use relative paths, which is required when hosted under a subdirectory like `/<repo-name>/`.

### 2. Netlify

**Best for:** Automatic deploys on push, instant rollbacks, preview deploys for PRs.

**Setup:**

1. Connect your GitHub repository at [netlify.com](https://www.netlify.com/)
2. Configure build settings:
   - **Build command:** `cargo install trunk && rustup target add wasm32-unknown-unknown && trunk build frontend/index.html --release`
   - **Publish directory:** `frontend/dist`

**Or with a `netlify.toml` in the repo root:**

```toml
[build]
  command = "rustup target add wasm32-unknown-unknown && cargo install trunk && trunk build frontend/index.html --release"
  publish = "frontend/dist"

[[headers]]
  for = "/*.wasm"
  [headers.values]
    Content-Type = "application/wasm"
```

> **Note:** Netlify's free tier builds can be slow for Rust projects due to compilation time. Consider caching `target/` via a plugin or using CI to build and deploy the artifacts instead.

### 3. Vercel

**Best for:** Fast global CDN, automatic HTTPS, preview deployments per PR.

**Setup:**

Since Vercel doesn't natively support Rust builds, use GitHub Actions to build and deploy:

1. Build the WASM output in CI (same as the GitHub Pages build step)
2. Deploy the `frontend/dist/` directory using the [Vercel CLI](https://vercel.com/docs/cli):

```yaml
# In your GitHub Actions workflow after the build step:
- name: Deploy to Vercel
  run: npx vercel --prod --token=${{ secrets.VERCEL_TOKEN }} frontend/dist
```

**Alternatively**, commit the built `frontend/dist/` and point Vercel at it with a `vercel.json`:

```json
{
  "outputDirectory": "frontend/dist"
}
```

### 4. Cloudflare Pages

**Best for:** Extremely fast global CDN (300+ edge locations), generous free tier, no build minute limits on free plan.

**Setup:**

1. Go to [Cloudflare Pages](https://pages.cloudflare.com/)
2. Connect your GitHub repository
3. Configure:
   - **Build command:** `curl https://rustwasm.github.io/wasm-pack/installer/init.sh -sSf | sh && rustup target add wasm32-unknown-unknown && cargo install trunk && trunk build frontend/index.html --release`
   - **Build output directory:** `frontend/dist`

**Or use GitHub Actions to build, then deploy:**

```yaml
- name: Deploy to Cloudflare Pages
  uses: cloudflare/pages-action@v1
  with:
    apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
    accountId: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
    projectName: ensemble-emulator
    directory: frontend/dist
```

### 5. Self-Hosted (Nginx / Static Server)

**Best for:** Full control, custom domain, integration with existing infrastructure.

After building with `trunk build frontend/index.html --release`, copy `frontend/dist/` to your web server.

**Nginx example:**

```nginx
server {
    listen 80;
    server_name ensemble.example.com;
    root /var/www/ensemble;

    location / {
        try_files $uri $uri/ /index.html;
    }

    # Correct MIME type for WASM files
    types {
        application/wasm wasm;
    }

    # Cache WASM and JS (they have content hashes in filenames)
    location ~* \.(wasm|js)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}
```

**Quick local testing** (no install needed):

```bash
# Python
cd frontend/dist && python3 -m http.server 8080

# Node.js
npx serve frontend/dist

# Rust
cargo install basic-http-server
basic-http-server frontend/dist
```

> **Important:** The server must serve `.wasm` files with `Content-Type: application/wasm`. Most modern servers do this by default, but verify if you encounter loading errors.

### 6. itch.io

**Best for:** Game-oriented distribution, built-in community, player discovery, optional pay-what-you-want pricing.

**Setup:**

1. Build with Trunk: `trunk build frontend/index.html --release`
2. Zip the contents of `frontend/dist/`
3. On [itch.io](https://itch.io/), create a new project:
   - **Kind of project:** HTML
   - **Upload:** the zip file
   - **Viewport dimensions:** 1024 × 768 (or your preferred size)
   - **SharedArrayBuffer:** Enable if needed for threading
4. Check "This file will be played in the browser"

**Or use the itch.io CLI:**

```bash
# Install butler (itch.io CLI)
# See: https://itch.io/docs/butler/

# Build and push
trunk build frontend/index.html --release
butler push frontend/dist username/ensemble-emulator:html5
```

### 7. Embedding in an Existing Webpage

To embed the emulator within a section of a larger webpage, use an `<iframe>`:

```html
<iframe
    src="https://your-hosted-emulator.example.com/"
    width="1024"
    height="768"
    style="border: none;"
    allow="cross-origin-isolated"
></iframe>
```

If you want to embed it directly (without an iframe), the Trunk build output can be loaded as a module. Adjust `index.html` to target a specific container `<div>` instead of `<body>`, though this requires changes to the `run_internal_wasm` function in `egui_frontend.rs` to find a specific element by ID rather than appending to `document.body()`.

## Important Considerations

### MIME Types
The server **must** serve `.wasm` files with `Content-Type: application/wasm`. This is the most common deployment issue — if the emulator fails to load, check this first.

### HTTPS
WebAssembly and IndexedDB (used for save data) require a **secure context** (HTTPS or localhost). All the hosting platforms above provide HTTPS by default.

### Cross-Origin Headers
If you encounter issues with `SharedArrayBuffer` (needed for some threading features), the server must set:

```
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
```

For Netlify, add to `netlify.toml`:
```toml
[[headers]]
  for = "/*"
  [headers.values]
    Cross-Origin-Opener-Policy = "same-origin"
    Cross-Origin-Embedder-Policy = "require-corp"
```

### Storage Persistence
The emulator uses **IndexedDB** for storing ROMs, savestates, and configuration in the browser. This data persists across sessions but:
- Users can clear it via browser settings
- Safari may evict data after 7 days of inactivity (in some configurations)
- Private/incognito browsing may limit or disable storage

### Asset Size
WASM binaries can be large. To minimize load times:
- Build with `--release` (enables optimizations)
- Consider adding `wasm-opt` for further size reduction (Trunk supports this via `[tools]` in `Trunk.toml`)
- Enable gzip/brotli compression on your server (all platforms above do this automatically)

### Browser Compatibility
eframe's WASM backend uses **WebGPU** with a **WebGL2 fallback**. This works in all modern browsers:
- Chrome 113+ (WebGPU native)
- Firefox 120+ (WebGPU behind flag, WebGL2 fallback)
- Safari 17+ (WebGPU support)
- Edge 113+ (Chromium-based, same as Chrome)
